## 2. Software Engineering

### Overview

- **Systematic approach** to designing, developing, testing, and maintaining software
- **Collaboration tools** help teams coordinate work, avoid conflicts, and stay organized
- **GitLab** plays a central role in how teams build software

---

### Software Development Lifecycle (SDLC)

#### 1. Planning Phase

> In the planning phase, the team establishes the project's goals and determines the scope of the software. They create a timeline with deadlines, along with a budget and resource allocation. This phase is crucial for custom software, as it helps to identify the unique requirements of the business. Understanding these needs allows for the development of a clear path for the project.

| Key Activities | Deliverables | Critical For |
|----------------|--------------|--------------|
| Timeline creation, budget estimation, resource allocation | Project roadmap, scope document | Custom software identification of unique business requirements |

#### 2. Requirements Analysis Phase

> During the requirements analysis phase, the team gathers detailed information about what users expect from the software. This is typically done through interviews and surveys to collect genuine feedback. A Requirements Specification Document can be created, outlining all the gathered information. This phase ensures that the software will include the specific features necessary, making it truly customised.

- Key Activities: User interviews, surveys, feedback collection
- Deliverables: Requirements Specification Document (RSD)
- Goal: Ensure software includes all necessary custom features

#### 3. Design Phase

> In the design phase, the software's architecture is developed. This includes creating diagrams that illustrate how data will flow and mock-ups of the user interface. A Software Design Document (SDD) can also be produced to detail these designs. For custom software, this phase focuses on specific design elements, ensuring that all components work seamlessly together.

| Key Activities | Deliverables | Focus |
|----------------|--------------|-------|
| Architecture design, data flow diagrams, UI/UX mockups | Software Design Document (SDD), technical architecture | Ensuring all components integrate seamlessly |

#### 4. Development Phase

> During the development phase, the actual code for the software is written. Code reviews and internal testing are important at this stage to ensure that everything is functioning correctly. This phase is vital for custom software, as it implements unique specifications.

- Key Activities: Coding, peer reviews
- Best Practices: Code review before merge requests
- Focus: Implementing unique specifications correctly

#### 5. Testing Phase

> In the testing phase, the team conducts thorough checks to identify any bugs or issues within the software. Various types of tests, including unit testing, security testing, and integration testing, are utilised to confirm that everything is working properly. This phase ensures that the software integrates effectively and functions as intended, providing peace of mind.

| Test Type | Purpose |
|-----------|---------|
| Unit Testing | Verify individual components |
| Security Testing | Identify vulnerabilities |
| Integration Testing | Ensure component compatibility |

#### 6. Deployment Phase

> The deployment phase involves launching the software to the end-users. It is also important to provide user training and to ensure that the new software works well with any existing systems. This phase is essential for custom software, as it helps users adapt to the new tools effectively.

- Key Activities: Software launch, user training, system integration verification
- Considerations: Compatibility with existing systems

#### 7. Maintenance Phase

> In the maintenance phase, ongoing support and updates are provided for the software. Addressing user feedback and making regular updates is essential to keep the software running smoothly. For custom software, this phase is crucial as it allows for adaptations to changing business needs, ensuring that the software remains relevant and effective.

- Key Activities: Ongoing support, bug fixes, feature updates
- Importance: Adapting to changing business needs

### GitLab Workflow Integration

#### 1. Plan & Design

Issues & Milestones: Use these to organize requirements and roadmap your project.

    gitlab issue create --title "Requirement X"

#### 2. Code & Build

Git Repos & Merge Requests: Use branching for development and MRs for collaborative code reviews.

    git checkout -b feature/new-feature
    git commit -m "Add new feature"
    git push origin feature/new-feature

#### 3. Test

GitLab CI: Automatically run scripts to build and test your application every time code is pushed.

    test:
      script:
        - npm test
        - npm run lint

#### 4. Release & Deploy

Continuous Deployment (CD): Automatically push passed code to staging or production environments.

| Environment | Status |
|-------------|--------|
| Staging | Review & QA |
| Production | Live deployment |

#### 5. Monitor & Maintain

Incident Management: Track bugs and monitor application performance, which feeds back into new "Plan" issues.

    Monitoring -> Bugs Identified -> New Issues Created -> Plan Phase

---

### Summary Table

| Phase | GitLab Feature | Purpose |
|-------|---------------|---------|
| Plan | Issues, Milestones | Requirement tracking |
| Code | Repositories, MRs | Collaborative development |
| Test | GitLab CI | Automated testing |
| Deploy | CD Pipelines | Continuous deployment |
| Monitor | Incident Management | Performance tracking |

## 3. SDLC and GitLab Integration

### SDLC Phases & GitLab Counterparts

| Phase | GitLab Feature | Key Components |
|-------|----------------|----------------|
| Design | Issues/Milestones | Issues, Milestones, Issue Boards |
| Development | Git Repos | Branching, Feature Work, Merge Requests |
| Testing | CI Pipelines | Auto-Testing, Different pipelines for actions |
| Deployment | CD Pipeline | Staging Environment, Production |
| Maintenance | Maintenance Tools | Error Tracking, User Feedback, Loop to Issues |

---

### 1. Design Phase - GitLab Issues & Milestones

> You might wonder: How does a simple 'Issue' cover an entire planning phase? Think of a GitLab Issue as the 'Project Manager' of that specific feature. It starts as a Description of what we need. Then, we use Labels to set the priority and Assignees to decide who is responsible. Finally, we link it to a Milestone to set a deadline and track it on a Board to see our progress. By the time a developer starts coding, the 'Planning' is already finished because everything they need is documented right there in the Issue.

Issue Lifecycle:
    Create Issue -> Add Description -> Set Labels -> Assign -> Add to Milestone -> Track on Board

---

### 2. Development Phase - Git Repositories

> Once the plan is set, we move to the Development phase. Developers create branches in the Git Repository to write their code. This ensures that the experimental work is separated from the stable production code. We use Merge Requests for peer review and collaboration.

Branching Strategy:
    main (production-ready)
    feature/login-system
    feature/user-dashboard
    hotfix/bug-fix-123

Merge Request Process:
    Feature Branch -> Push Code -> Create MR -> Code Review -> Tests Run -> Approvals -> Merge to Main

---

### 3. Testing Phase - CI Pipelines

> The most critical part of the SDLC is Testing. In GitLab, we don't just 'hope' the code works. CI Pipelines automatically run tests. If the pipeline turns red, the SDLC stops until the code is fixed.

    stages:
      - build
      - test
      - security
      - deploy

    build:
      stage: build
      script: npm install && npm run build

    test:
      stage: test
      script: npm test

---

### 4. Deployment Phase - CD Pipelines

> After testing comes the Release. GitLab automates this through Continuous Delivery (CD). We can push the code to a staging environment for final checks and then to production with a single click.

Deployment Pipeline:
    Tests Pass -> Deploy to Staging -> QA Approval -> Manual Trigger -> Deploy to Production -> Verify

---

### 5. Maintenance Phase - Monitoring & Feedback

> The SDLC is a cycle, not a line. We monitor the software in production using Analytics. When a user finds a bug, we create a new Issue, and the entire SDLC cycle starts all over again.

Feedback Loop:
    Production -> Monitor -> Errors/Bugs Found -> Create Issue -> Plan Phase -> Develop -> Test -> Deploy

---

### Key Takeaways

1. Complete Coverage: GitLab provides tools for every SDLC phase
2. Automation: CI/CD reduces manual errors and speeds delivery
3. Traceability: Every change linked to issues and requirements
4. Collaboration: Merge requests enable team coordination
5. Continuous Loop: Maintenance feeds back into planning seamlessly

## 4. Creating an Issue

### Overview

- Discrete work item used to plan, track, and discuss tasks within a project
- Go to Plan > Work Items
- Issue Board: Used to plan, organize, and visualize issues
- Documentation: Use wiki for the documentation of whole project

---

### What is an Issue?

A discrete work item used to plan, track, and discuss tasks within a project. Each issue represents a specific task, bug, feature request, or improvement that needs to be completed.

---

### Key Features

| Feature | Purpose |
|---------|---------|
| Plan > Work Items | Navigate to issue management area |
| Issue Boards | Visualize workflow (To Do, In Progress, Done) |
| Wiki | Project-wide documentation |

---

### Issue Board Workflow

1. To Do - New issues start here
2. In Progress - When work begins
3. Done - Completed and verified

---

### Documentation Best Practices

- Use Wiki for whole project documentation
- Keep docs organized and version-controlled alongside code
- Include setup instructions, architecture diagrams, and API docs

---

### Quick Reference

**Navigation Path:**
Project -> Plan -> Work Items -> New Issue

**Issue Properties:**
- Title: Brief description of task
- Assignee: Who will do the work
- Labels: Priority, type, category tags
- Milestone: Deadline/project phase
- Description: Detailed requirements

## 5. Installing Git

### Overview

- Open Company portal: In start menu, search for "company portal" / "Unternehmensportal"
- Search for Git and install
- Open Git Bash from start menu

---

### Git Configuration

Configure Git user name and email to label commits

| Command | Purpose |
|---------|---------|
| git config --global user.name "Your Name" | Set user name |
| git config --global user.email "you@domain.com" | Set email |
| git config --list | View all configuration parameters |

---

### Step-by-Step Installation

1. **Search** for company portal in Start Menu
2. **Find** Git application in the portal
3. **Install** the software
4. **Open** Git Bash from Start Menu after installation
5. **Configure** your identity with commands above

---

### Example Commands

    git config --global user.name "John Doe"
    git config --global user.email "john@example.com"
    git config --list

---

### Verification

After configuration, verify your settings:

    $ git config --list
    user.name=John Doe
    user.email=john@example.com

---

*Configuration is stored globally for all Git repositories on your machine*

## 6. Start a New Repository

### Overview

- Initialize the Repository: git init
- Create/modify the files
- Add Files: git add .
- Commit the Initial Version: git commit -m "message"
- Create a Remote Repository
- Connect Local to Remote: git remote add origin <repository_url>
- Push to Remote: git push origin main

---

### Step 1: Initialize the Repository

**Command:** git init

Marks the directory as a Git project

    $ mkdir my-project
    $ cd my-project
    $ git init
    Initialized empty Git repository in /path/to/my-project/.git/

---

### Step 2: Create/Modify the Files

Create your project files (e.g., index.html, app.js, README.md)

    $ touch index.html
    $ touch app.js
    $ echo "# My Project" > README.md

---

### Step 3: Add Files

**Command:** git add .

Adds all files in the current directory to the staging area

    $ git add .
    $ git status
    On branch main
    Untracked files:
      index.html
      app.js
      README.md
    Changes to be committed:
      new file: index.html
      new file: app.js
      new file: README.md

---

### Step 4: Commit the Initial Version

**Command:** git commit -m "message"

Saves a snapshot of your initial project files

    $ git commit -m "Initial commit"
    [main (root-commit) abc1234] Initial commit
    3 files changed, 10 insertions(+)
    create mode 100644 README.md
    create mode 100644 app.js
    create mode 100644 index.html

---

### Step 5: Create a Remote Repository

Go to GitLab and create a new, empty project

Important: Don't initialize it with a README or license file

| Setting | Value |
|---------|-------|
| Project Name | my-project |
| Visibility | Private/Public |
| README | No |
| License | None |
| .gitignore | None |

---

### Step 6: Connect Local to Remote

**Command:** git remote add origin <repository_url>

Tells Git where to push your changes

    $ git remote add origin https://gitlab.com/username/my-project.git
    $ git remote -v
    origin  https://gitlab.com/username/my-project.git (fetch)
    origin  https://gitlab.com/username/my-project.git (push)

---

### Step 7: Push to Remote

**Command:** git push origin main

Copies your local repository to the remote repository

    $ git push origin main
    Enumerating objects: 5, done.
    Counting objects: 100% (5/5), done.
    Delta compression using up to 8 threads
    Compressing objects: 100% (3/3), done.
    Writing objects: 100% (3/3), 295 bytes | 295.00 KiB/s, done.
    Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
    To https://gitlab.com/username/my-project.git
     * [new branch]      main -> main

---

### Complete Workflow Summary

    git init → Create Files → git add . → git commit → 
    git remote add origin URL → git push origin main

---

### Quick Reference Table

| Action | Command |
|--------|---------|
| Initialize | git init |
| Stage All Files | git add . |
| Commit | git commit -m "message" |
| Add Remote | git remote add origin <URL> |
| Verify Remote | git remote -v |
| Push to Remote | git push origin main |

## 7. Clone Existing Repository

### Overview

- Create a local copy of remote repository
- Open the remote project, find its URL
- Open Git Bash in the local directory where you want to create a local copy
- Use the command: git clone <project_url>

---

### What is Cloning?

Cloning creates a local copy of an existing remote repository on your machine. This allows you to work on the project locally and sync changes with the remote repository.

---

### Step-by-Step Guide

#### Step 1: Find the Remote URL

Open the remote project on GitLab and locate the clone URL

    https://gitlab.com/username/project-name.git

| Button | Purpose |
|--------|---------|
| Code | View clone options |
| HTTPS | Secure protocol |
| SSH | SSH key authentication |

---

#### Step 2: Navigate to Local Directory

Open Git Bash in the directory where you want to create the local copy

    $ cd ~/Documents/GitProjects
    $ pwd
    /c/Users/username/Documents/GitProjects

---

#### Step 3: Clone the Repository

**Command:** git clone <project_url>

    $ git clone https://gitlab.com/username/project-name.git
    Cloning into 'project-name'...
    remote: Enumerating objects: 150, done.
    remote: Counting objects: 100% (150/150), done.
    remote: Compressing objects: 100% (98/98), done.
    remote: Total 150 (delta 42), reused 130 (delta 30)
    Receiving objects: 100% (150/150), 125 KiB | 1.09 MiB/s, done.
    Resolving deltas: 100% (42/42), done.

---

#### Step 4: Verify the Clone

Navigate into the cloned directory and check the status

    $ cd project-name
    $ git status
    On branch main
    Your branch is up to date with 'origin/main'.
    nothing to commit, working tree clean

---

### Working with Cloned Repositories

#### Common Commands

| Command | Purpose |
|---------|---------|
| git pull | Fetch and merge latest changes |
| git status | Check current state |
| git log | View commit history |
| git branch -a | List all branches |

---

### Troubleshooting

| Issue | Solution |
|-------|----------|
| Permission denied | Configure SSH keys or use HTTPS credentials |
| Connection timeout | Check internet connection |
| Repository not found | Verify URL and access permissions |

---

### Complete Workflow Summary

    Find URL → Open Git Bash → Navigate to Directory → 
    git clone <URL> → cd project-name → Start Working

---

### Key Points

1. **Local Copy**: Full repository with all history and branches
2. **Remote Tracking**: Automatically tracks origin remote
3. **Ready to Work**: Can immediately make changes and commits
4. **Sync Changes**: Use git pull and git push to sync with remote

## 8. Branching and Merge Request

### Overview

- **Branch**: A moveable pointer to a commit
- **Work in Parallel**: Multiple team members can work on different tasks simultaneously without overwriting each other

---

### Key Git Commands

| Command | Purpose | Example |
|---------|---------|---------|
| git branch [name] | Create a new branch | git branch feature-login |
| git switch [name] | Switch to a branch | git switch feature-login |
| git merge [name] | Combine changes back to main | git merge feature-login |
| git branch | List all branches | git branch |
| git branch -a | List all branches including remote | git branch -a |

---

### What is a Merge Request?

A formal proposal to merge changes from a feature branch into main branch. It enables peer review, discussion, and approval before integrating code.

---

### The Workflow

    Push your branch and open an MR to summarize your changes
    Source Branch: The branch we want to merge
    Target Branch: The branch that your changes will be merged into
    Teammates discuss the code, suggest improvements

---

### Step-by-Step Merge Request Process

#### Step 1: Push to Test Branch

    $ git push origin feature-branch-name

#### Step 2: Create Merge Request

Navigate to GitLab and create a new merge request

| Field | Description |
|-------|-------------|
| Source Branch | The branch we want to merge (feature branch) |
| Target Branch | The branch that your changes will be merged into (main) |
| Name | Clear title of the change |
| Description | Summary of changes made |

#### Step 3: Discussion & Code Review

- Teammates discuss the code
- Suggest improvements
- Address feedback
- Push additional commits if needed

    $ git commit -am "Addressed review feedback"
    $ git push origin feature-branch-name

#### Step 4: Merge into Target Branch

After approval, merge the branch

    $ git merge main
    $ git push origin main

---

### Merge Request Best Practices

| Practice | Purpose |
|----------|---------|
| Small MRs | Easier to review and test |
| Clear Titles | Quickly understand the change |
| Descriptive Descriptions | Provide context for reviewers |
| Linked Issues | Connect to tracking system |
| Pass CI Pipeline | Ensure tests pass before merging |
| Request Reviewers | Get right people to review |

---

### Common Merge Request Statuses

| Status | Meaning |
|--------|---------|
| Open | MR created, awaiting review |
| In Progress | Work continuing, more commits expected |
| Approved | Ready to merge after final check |
| Merged | Changes integrated into target branch |
| Closed | MR rejected or no longer needed |
| Conflicts | Requires conflict resolution before merge |

---

### Complete Branching Workflow

    git branch feature-name
    git switch feature-name
    # Make changes
    git add .
    git commit -m "Feature implemented"
    git push origin feature-name
    # Open MR on GitLab
    # Wait for review
    # Merge to main

---

### Key Benefits

1. **Isolation**: Work without affecting main branch
2. **Collaboration**: Team review improves code quality
3. **Traceability**: Track what changed and why
4. **Safety**: Tests run before merging prevents bugs
5. **Documentation**: MR serves as change log

## 9. Deployment

### Overview

- **CI (Continuous Integration)**: An automated pipeline that builds, tests, and deploys code every time we make a change, ensuring speed and quality while eliminating manual errors
- **CD (Continuous Delivery)**: Triggers after CI to deploy code from Staging to Production for the end user

---

### What is CI/CD?

| Term | Full Name | Purpose |
|------|-----------|---------|
| CI | Continuous Integration | Automate build and testing on every code change |
| CD | Continuous Delivery/Deployment | Automate deployment to staging/production |

---

### CI Pipeline Workflow

    Code Commit → Build → Test → Security Scan → Pass/Fail

Every time code is pushed, the CI pipeline runs automatically:

| Stage | Action |
|-------|--------|
| Build | Compile code, install dependencies |
| Test | Run unit tests, integration tests |
| Security | Scan for vulnerabilities |
| Report | Generate test coverage reports |

---

### CD Pipeline Workflow

    CI Success → Deploy to Staging → Manual Review → Deploy to Production

After CI passes, CD takes over:

| Environment | Purpose | Trigger |
|-------------|---------|---------|
| Staging | Final QA & Testing | Automatic after CI pass |
| Production | Live User Access | Manual approval or automatic |

---

### GitLab CI/CD Example (.gitlab-ci.yml)

    stages:
      - build
      - test
      - security
      - deploy

    build:
      stage: build
      script:
        - npm install
        - npm run build

    test:
      stage: test
      script:
        - npm test

    security:
      stage: security
      script:
        - npm audit

    deploy:
      stage: deploy
      script:
        - ./deploy.sh
      environment: production

---

### CI vs CD Comparison

| Feature | CI | CD |
|---------|-----|-----|
| Focus | Code Quality | Deployment Automation |
| When | On Every Commit | After CI Passes |
| Output | Test Reports | Deployed Application |
| Manual Step | No | Optional (Staging) |
| Risk Level | Low | Medium (Production) |

---

### Benefits of CI/CD

1. **Speed**: Faster releases with automation
2. **Quality**: Consistent testing prevents bugs
3. **Reliability**: Repeatable processes reduce errors
4. **Visibility**: Clear status for all team members
5. **Feedback**: Immediate notification on failures

---

### Complete Deployment Flow

    Developer Commits Code
          ↓
    Git Push to Remote
          ↓
    GitLab CI Pipeline Starts
          ↓
    Build → Test → Security Scan
          ↓
    If Pass → Deploy to Staging
          ↓
    QA Approval / Manual Review
          ↓
    If Approved → Deploy to Production
          ↓
    Users Access New Features

---

### Key Commands

| Command | Purpose |
|---------|---------|
| git push | Trigger CI pipeline |
| git merge | Trigger CD pipeline |
| git log | View deployment history |

---

### Monitoring Deployment

- Check pipeline status in GitLab CI tab
- Monitor application logs in production
- Track error rates and performance metrics
- Feedback loop back to Issues/Bugs
